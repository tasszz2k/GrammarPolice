# Signing certificate runbook

GrammarPolice releases are signed with a long-lived self-signed code-signing
certificate. The certificate is reused across every release on purpose, so
that the macOS Accessibility grant survives upgrades. This document is the
operator's runbook for everything related to that certificate.

## The three secrets

The release workflow ([`.github/workflows/release.yml`](../.github/workflows/release.yml))
needs three GitHub Actions secrets in `tasszz2k/GrammarPolice`:

| Secret | What it is | Backup needed? |
|--------|------------|----------------|
| `MACOS_CERT_P12_BASE64` | Base64 of the PKCS#12 bundle (private key + cert), generated by `scripts/generate-signing-cert.sh`. | Yes - this is the project's de-facto Developer ID. |
| `MACOS_CERT_PASSWORD` | Password that decrypts the p12. | Yes - the p12 is unusable without it. |
| `MACOS_KEYCHAIN_PASSWORD` | Random password for the temporary keychain inside the CI runner. | No - regenerate any time. |

Only the first two need to be backed up. The keychain password is transient
and can be regenerated on demand without affecting anything else.

## Why the cert must stay stable

macOS TCC keys the Accessibility grant by the application's signing identity.
For non-Apple-anchored signatures (i.e. anything that is not a Developer ID
issued by Apple), TCC matches against the bundle identifier and the
**certificate leaf hash** that appears in the designated requirement:

```
designated => identifier "com.tasszz2k.GrammarPolice"
              and certificate leaf = H"<sha1 of the cert>"
```

Because we use the same p12 across every release, the cert leaf hash stays
constant, the designated requirement stays constant, and TCC keeps the
existing grant.

Generating a new p12 produces a brand-new keypair with a different leaf hash.
TCC then sees the next release as a different app and drops the existing
Accessibility grant. Every existing user has to re-grant once.

## Backup recommendation

Store both files in two independent places:

1. A password manager (1Password, Bitwarden, iCloud Keychain, etc.) - attach
   the actual `gp.p12` file as an attachment and store the cert password as
   a secure note on the same item.
2. An offline encrypted copy - encrypted USB, or an encrypted DMG you keep
   on a drawer USB drive. Easiest:

   ```bash
   hdiutil create -encryption AES-256 -volname GPCertBackup \
     -size 10m -fs APFS GPCertBackup.dmg
   ```

The CI runner only ever sees the secrets in GitHub Actions, never these
backups.

## PKCS#12 format gotcha (`-legacy`)

The p12 must be created with `openssl pkcs12 -export -legacy`, not the
OpenSSL 3.x default. The macOS `security` command's PKCS#12 importer only
understands the older `pbeWithSHA1And40BitRC2-CBC` + HMAC-SHA-1 encryption.
A modern p12 (PBES2 + AES-256 + HMAC-SHA-256) decodes successfully with
`openssl pkcs12 -info` but fails the keychain import in CI with the
misleading error:

```
security: SecKeychainItemImport: MAC verification failed during PKCS12 import (wrong password?)
```

The password is fine in that case; the algorithm is not. `scripts/generate-signing-cert.sh`
already uses `-legacy`, but if you ever export a p12 manually (e.g. when
migrating an existing private key from Keychain Access), make sure it is in
the legacy format. To convert an OpenSSL 3.x p12 to the legacy format
without changing the underlying key/cert:

```bash
OLD=path/to/modern.p12
OLD_PASS='...'
NEW_PASS=$(openssl rand -hex 16)
WORK=$(mktemp -d)

openssl pkcs12 -in "$OLD" -passin pass:"$OLD_PASS" -nocerts -nodes -out "$WORK/key.pem"
openssl pkcs12 -in "$OLD" -passin pass:"$OLD_PASS" -clcerts -nokeys  -out "$WORK/cert.pem"

openssl pkcs12 -export -legacy \
  -inkey "$WORK/key.pem" -in "$WORK/cert.pem" \
  -name "GrammarPolice Self-Signed" \
  -out "$WORK/legacy.p12" \
  -passout pass:"$NEW_PASS"

# Use $WORK/legacy.p12 + $NEW_PASS as the new secrets.
```

The cert leaf hash is preserved (same key + same x509 cert, only the p12
container differs), so this kind of conversion is **not** a rotation -
existing users keep their Accessibility grant.

## When you need to rotate the certificate

Rotate only when one of these is true:

- The p12 or its password has been lost.
- The p12 has been leaked or might be on a compromised machine.
- The certificate is approaching its 10-year expiry (current cert is valid
  for 10 years from generation).
- You are migrating to an Apple Developer ID (in which case follow the
  Developer ID migration section instead).

Rotating has a one-time cost: every existing user is re-prompted for
Accessibility on the first release after the rotation. After that, things
return to normal.

## Rotation procedure

```bash
cd /path/to/GrammarPolice

# 1. Generate a new cert and back it up locally.
./scripts/generate-signing-cert.sh
# The script prints the three values and copies the base64 to clipboard.
```

Then either upload via the GitHub UI at
`https://github.com/tasszz2k/GrammarPolice/settings/secrets/actions`,
or via CLI without printing the values to your terminal:

```bash
# Assumes scripts/generate-signing-cert.sh wrote gp.p12 + cert_password.txt
# to ~/.GrammarPoliceCert/ (the helper does this when run via the operator
# pipeline).
REPO=tasszz2k/GrammarPolice
gh secret set MACOS_CERT_P12_BASE64    --repo "$REPO" < <(base64 -i ~/.GrammarPoliceCert/gp.p12 | tr -d '\n')
gh secret set MACOS_CERT_PASSWORD      --repo "$REPO" < ~/.GrammarPoliceCert/cert_password.txt
gh secret set MACOS_KEYCHAIN_PASSWORD  --repo "$REPO" < <(openssl rand -hex 16 | tr -d '\n')

gh secret list --repo "$REPO" | grep MACOS_
```

Verify the secrets show up in `gh secret list` with a recent timestamp,
**back up the new p12 and password** to your password manager and offline
copy, then cut the next release as normal (release-please will pick it up
automatically on the next merge to `main`).

## What to communicate to users when you rotate

In the release notes for the first release after a rotation, add a note like:

> This release was signed with a new code-signing certificate. macOS will
> ask once for Accessibility permission again on first launch after the
> upgrade. After granting it, future upgrades will not re-prompt.

This avoids users thinking the app is broken when TCC drops the grant.

## Verifying a built `.app` matches the expected identity

To check whether a build will keep the Accessibility grant for existing
users, compare the designated requirement with a previous release:

```bash
codesign -d -r- /Applications/GrammarPolice.app
# expect:
#   designated => identifier "com.tasszz2k.GrammarPolice"
#                 and certificate leaf = H"<same hash as before>"
```

If the cert leaf hash matches the previous release, TCC will accept the
upgrade. If it differs, you have rotated the cert (intentionally or not).

## Migrating to an Apple Developer ID (the proper fix)

If at some point you decide to pay for the Apple Developer Program, swap the
self-signed cert for a `Developer ID Application` certificate:

1. Export the Developer ID cert + private key from Keychain Access as a p12.
2. Replace `MACOS_CERT_P12_BASE64` and `MACOS_CERT_PASSWORD` with values
   derived from the new p12 (same procedure as rotation above).
3. Replace `--sign "GrammarPolice Self-Signed"` in
   [`.github/workflows/release.yml`](../.github/workflows/release.yml) with
   `--sign "Developer ID Application: <Your Name> (<TEAMID>)"`.
4. Add a notarization step after signing:

   ```yaml
   - name: Notarize
     run: |
       ditto -c -k --keepParent "${APP}" "${APP}.zip"
       xcrun notarytool submit "${APP}.zip" \
         --apple-id "$APPLE_ID" \
         --team-id "$APPLE_TEAM_ID" \
         --password "$APPLE_APP_SPECIFIC_PASSWORD" \
         --wait
       xcrun stapler staple "${APP}"
   ```

   This requires three additional secrets: `APPLE_ID`,
   `APPLE_TEAM_ID`, `APPLE_APP_SPECIFIC_PASSWORD`.

This counts as a rotation for TCC purposes (different cert leaf hash), so
existing users will be re-prompted once. From then on, Gatekeeper stops
flagging the app on first launch (no more `xattr -cr` needed) and the cert
is renewed by Apple every year.
